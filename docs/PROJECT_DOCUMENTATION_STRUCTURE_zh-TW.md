## 目的

本文件定義 HANSOME 專案的**整體文件架構**。

內容包括：

- 每一份正式專案文件；
- 各文件的職責與非職責；
- 版本與生命週期規則；
- 未來貢獻者應如何維護文件。

目標是**避免**把遊戲規則、NFT 設計、經濟、安全分析、路線圖規劃與實作標準混在同一份檔案裡。

**本檔不是遊戲規則文件。**  
不得新增、修改或重新詮釋玩法、獎勵、財庫參數或 NFT 機制效果。既有已鎖定的設計文件，在其主題上仍為權威。

## 1. 核心遊戲規則

### 檔名

`01_Core_Game_Rules_v1.1.md`

### 目的

**正式遊戲規格**。

只包含理解 HANSOME「一日如何遊玩與結算」所需的**玩法**與玩家向系統規則。

### 必須包含

- 遊戲總覽
- 財庫（作為遊戲系統概念：獎勵來自遊戲財庫；不可增發）
- 每日流程
- Commit
- Reveal
- Settlement
- Claim
- 地圖／地點
- 羊駝能力
- 美洲獅能力
- 獎池
- 狩獵
- Token Sink
- 財庫更新規則（含排放降階等遊戲經濟行為）
- 邊界情況

### 不得包含

- Solidity 或其他實作程式碼
- 資料夾結構、CI、編碼規範
- 美術流程、色票、渲染規格
- 攻擊清單與修補（改連至附錄 A）
- 蒙地卡羅程序或證明長文（改連至附錄 B）
- 路線圖日期或行銷里程碑
- 重複的長篇安全論文

### 規則

- 該版本**發布後鎖定**（例如 v1.1）。
- **僅能**透過版本升級更新（`v1.2`、`v1.3`…）——禁止在仍標示 v1.1 時默默改規則。
- **禁止**混入實作筆記。
- 定義現行遊戲的參數表應置於此（或以單一權威表形式由本檔規範性引用）。

### 與 GDS 的關係

歷史文件 `HANSOME_GDS_v1.1_*.md` 為完整設計凍結。  
`01_Core_Game_Rules_v1.1.md` 應成為對齊該凍結的**精簡玩法抽取版**。在 `01_…` 存在前，以 GDS v1.1 為玩法過渡權威。

### 檔名

`02_Appendix_A_Security_and_Fair_Play.md`

### 目的

針對核心遊戲規則所描述系統的安全審查與公平遊玩指引。

### 必須包含

- Commit 機密性
- 弱 salt 風險
- 重放保護
- Reveal 洩漏
- VRF／隨機數完整性期望

- 前端資訊洩漏
- 鏈下串謀
- 公平遊玩檢查清單

### 不得包含

- 新玩法機制
- 獎池比例、權重或特質效果之變更
- 美術方向
- 路線圖承諾

### 規則

- **不修改玩法。**
- 可建議產品、UX 或營運控管。
- 若安全發現需要改規則，應對**核心遊戲規則**提出新版本——不得只在本附錄內偷偷修補規則。

`04_NFT_Design_Bible_v1.0.md`

### 目的

正式 **NFT 美術與系列呈現**規格。

### 必須包含

- 系列總覽
- 原型（Archetypes）
- 特質（視覺／目錄意義）
- 特質規則（構圖、圖層互斥等）

- 設計哲學
- 美術製作流程
- 審稿流程

### 不得包含

- 遊戲公式（懲罰率、獎池拆分、結算數學）
- 智能合約邏輯
- 排放控制器方程式

### 規則

- **不含遊戲公式。**
- **不含智能合約邏輯。**
- **僅 NFT 規格**（創意 + metadata + 流程）。
- 當原型對應玩法類型（例如 Guardian）時，以**引用**核心遊戲規則表述對應關係——勿在此重推戰鬥數學。

美術聖經自 **v1.0** 起，僅可就視覺／metadata 事項獨立迭代。任何改變鏈上特質**機制身分**的變更，必須進入核心遊戲規則的版本討論。

## 版本政策

### 文件版本識別

每份正式文件應具備：

- 適用時的**檔名版本**（`_v1.1`、`_v1.0`）；
- 文內 **SemVer 風格**或政策版本欄位；
- **狀態**（見生命週期）。

### 主／次版本（示例）

| 標示 | 意義 |
|------|------|
| `v1.0` | 該文件類別的首個核准基線 |
| `v1.1` | 同世代內之相容演化或修正集合 |
| `v1.2` | 下一世代遞增；**僅**核心遊戲規則可在此包含規則變更 |

對**核心遊戲規則**而言，發布後要改鎖定玩法，唯一正當途徑是 `v1.1` → `v1.2`。

對**NFT 設計聖經**，`v1.0` → `v1.1` 可涵蓋不改變機制身分的美術／metadata 修訂。

### 修補版本

| 標示 | 意義 |
|------|------|
| `v1.1.1` | 錯字、澄清、連結修復、格式——**無**語意規則變更 |

若「修補」會改變獎勵、特質或結算，那就**不是修補**；應升 minor／major 並走審核。

### 工程版本 vs 文件版本

- 文件版本描述**規格**。
- 軟體發行標籤（若有）必須標明所實作之文件版本。
- 對應關係記在開發聖經——不要寫進核心遊戲規則敘事本文。

## 文件生命週期

每份正式文件使用下列狀態之一：

| 狀態 | 意義 |
|------|------|
| **Draft** | 進行中；不具權威 |
| **Review** | 已送交設計／安全／美術／工程審核 |
| **Approved** | 經指定負責人接受；可營運使用 |
| **Locked** | 針對具名版本凍結；修改需新版本，或零語意變更之書面修補 |

### 轉換規則

1. 作者認為可接受批評時：`Draft` → `Review`。
2. 負責人簽核後：`Review` → `Approved`。
3. 對更廣團隊或公眾具約束力時：`Approved` → `Locked`。
4. 變更已 `Locked` 的核心遊戲規則：必須開**新版本檔**（例如 `01_Core_Game_Rules_v1.2.md`），並在舊檔加棄用橫幅。

### 擁有者（邏輯角色）

| 文件 | 預設擁有角色 |
|------|------|
| 核心遊戲規則 | 遊戲設計負責人 |
| 安全與公平遊玩 | 安全／設計共同擁有 |
| 數學與經濟驗證 | 經濟／設計共同擁有 |
| NFT 設計聖經 | 美術總監 |
| 專案路線圖 | 專案負責人 |
| 開發聖經 | 工程負責人 |
| 文件架構說明 | 專案負責人 + 文件維護者 |

## 交叉引用規則

1. **優先引用，勿複製貼上。** 必要時可短引不變量；勿維護兩套完整公式。
2. 使用穩定錨點：文件 ID + 章節標題 + 版本。
3. 引用示例：

   > 見核心遊戲規則 v1.1 § 狩獵池分配。

4. 附錄僅可為**驗證**而重複公式，且必須標明正在驗證的核心規則章節。
5. NFT 設計聖經可列特質*名稱*與數量；機制效果連出至核心遊戲規則。
6. 路線圖可寫「Season 1 開啟狩獵賽季」，無需重寫懲罰係數。

## 文件原則

1. **一份文件一件職責。**  
   若一節同時適合兩根柱，拆分或把其中一方降為引用。

2. **不要重複資訊。**  
   重複會造成分叉；分叉造成不公平遊戲與失敗稽核。

3. **優先引用，勿複製貼上。**  
   每條規則只保留一處權威陳述。

4. **玩法與實作分離。**  
   核心遊戲規則不是程式風格或倉庫配置的垃圾場。

5. **NFT 設計獨立於遊戲機制。**  
   視覺聖經不得重定義戰鬥或獎勵。

6. **安全文件永不改玩法。**  
   它約束的是如何保護已約定的遊戲。

7. **數學驗證永不引入新機制。**  
   只檢查我們已約定的機制。

8. **路線圖是願景，不是規格。**
